# 反馈循环与附加要求规则

## 目的
累积地捕捉、组织和应用用户反馈，确保 AI 作者能够随着时间的推移从偏好和错误中“学习”，而无需重复提示。

## 文件结构
-   **位置**: `./feedback/` 目录（相对于项目根）。
-   **文件名**: `requirements.md` (单一事实来源)。
-   **归档**: `./feedback/archive/<YYYY-MM-DD>.md`（按归档流程移出的过期/被覆盖规则）。

## 条目元数据格式

每条规则前必须有一行 HTML 注释，承载机读元数据：

```markdown
<!-- id: R042  added: 2026-04-22  last_hit: 2026-04-22  source: user -->
- [ ] **风格强化**：修辞意象需更加色情、淫靡，通过环境（气味、光影）烘托背德氛围。
```

字段定义：
- `id`: 全局唯一标识（建议 `R<3位序号>`，递增不复用）。
- `added`: 该规则首次写入的日期（ISO 短格式）。
- `last_hit`: 最近一次"在生成中应用过该规则"的日期；每次草稿前的规则扫描可一并更新。
- `source`: `user` / `system` / `derived`（从多条合并而来）。
- 被覆盖时追加 `superseded_by: R<id>` 字段。

旧条目（无元数据）不必强制回填——只在下次修改它时顺手补上。

## 内容结构
`requirements.md` 文件必须按以下特定类别组织：

### 1. 风格与基调偏好 (Style & Tone Preferences)
*关于文本听起来应该是什么样子的正面约束。*
-   **首选 (Preferred)**: (例如，“在动作场景中使用简短有力的句子。”)
-   **避免 (Avoid)**: (例如，“不要用华丽的隐喻来形容简单的物体。”)

### 2. 角色塑造规则 (Characterization Rules)
*关于角色如何说话或行动的具体规则。*
-   **[角色名称]**: (例如，“总是用谜语说话”，“从不使用缩略语。”)

### 3. 格式与结构 (Formatting & Structure)
*输出的技术要求。*
-   (例如，“标题总是使用 Markdown 格式”，“场景之间用 `***` 分隔。”)

### 4. 负面约束 (Negative Constraints - "Don't" List)
*必须严格避免的特定套路、短语或模式。*
-   (例如，“没有‘从梦中醒来’的结局”，“避免以‘突然’开头的句子。”)
-   (用户反馈示例) “不要只描写交合处，而要遍览全身。”

### 5. 感官与情色密度 (Sensory & Erotic Density)
*关于肉戏、氛围渲染和感官细节的具体要求。*
-   **AV感设计 (AV Style)**: 视觉奇观、特写镜头、表演性张力。
-   **交互范围 (Interaction Scope)**: 全身互动（头发、背部、四肢），而非局限于性器官。
-   **语言风格 (Language)**: Dirty Talk 的密度与耻度控制。

### 6. 恐怖与惊悚密度 (Horror & Thriller Density)
*关于恐怖氛围、惊悚节奏和心理压迫的具体要求。*
-   **氛围营造 (Atmosphere Building)**: 环境描写、声音设计、光影对比。
-   **恐惧节奏 (Fear Pacing)**: 张弛有度，铺垫-暗示-爆发的节奏控制。
-   **未知恐惧 (Fear of Unknown)**: 留白艺术，不过度解释恐怖元素。
-   **心理惊悚 (Psychological Horror)**: 角色心理崩溃过程、不可靠叙事。

## 维护流程 (循环)
在每个写作任务或审查阶段结束时：
1.  **用户反馈**: 用户对生成的文本提供评论（例如，”这太煽情了”或”我喜欢对话但讨厌节奏”）。
2.  **分析**: AI 分析此反馈以提取*可通用的规则*。
3.  **更新**: AI 追加或修改 `requirements.md` 中的条目。
    -   *新增*: 分配新的 `id`，写入 `added` 与 `source: user` / `derived`。
    -   *冲突解决*: 比较新旧条目的 `added` 时间戳。新条目胜出后，**保留**旧条目但在其元数据中追加 `superseded_by: R<新id>`，并把旧条目移到 `./feedback/archive/<日期>.md`。
    -   *整合*: 如果多条规则相似，合并为一条 `source: derived` 的新规则，把被合并的几条加 `superseded_by` 后归档。
4.  **状态机联动**: 修改后更新 `./state.yaml.requirements_version` 为本次时间戳。

## 归档流程 (定期清理)
每完成一个 Chapter 后执行一次：
1.  扫描 `requirements.md` 所有条目的 `last_hit` 字段。
2.  对于”距今超过 N 章（建议 N=3）未被命中”的规则，移到 `./feedback/archive/<完成日期>.md`，并在原文件留一行链接：`<!-- archived: R042 -> archive/2026-04-22.md -->`。
3.  归档操作不删除任何内容，只迁移。需要时仍可从归档文件恢复。

## 使用规则
-   **初始化**: 在开始任何写作任务时，必须读取 `requirements.md` 文件。
-   **命中标记**: 草稿生成完成后，把本次实际应用过的规则的 `last_hit` 更新为当天日期（不必逐条精确，按章批量更新即可）。
-   **系统提示注入**: 该文件中的关键高优先级规则应注入系统提示 (System Prompt) 或写作生成的即时上下文窗口中。
